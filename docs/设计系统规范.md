# DreamBook 设计系统规范（对齐版）

## 1. 目标与边界

### 1.1 设计系统目标
- 统一产品视觉语言，让不同角色（产品、设计、客户端、后端、运营）在同一语义上对齐。
- 让 UI 可预测、可复用、可扩展，降低页面新增与重构成本。
- 在“梦境记录”场景中建立明确感知：安静、神秘、可沉浸、可信赖。

### 1.2 设计系统边界
- 设计系统负责：色彩、排版、间距、圆角、阴影、基础组件、主题模式规范。
- 业务模块负责：信息架构、交互流程、策略逻辑、内容生产。
- 任何业务页面不得绕开令牌直接硬编码核心视觉参数。

## 2. 设计哲学

### 2.1 现象层（用户感知）
- 用户在刚醒状态下使用，认知负担应低，信息层级要一眼可读。
- 页面需要传达“梦在被认真记录与理解”，而不是工具感表单。

### 2.2 本质层（系统原则）
- 单一真相源：颜色、排版、间距全部收敛到令牌层。
- 特例消失：新页面优先复用组件，而不是新增分支样式。
- 双模式同构：亮色与暗色用同一语义结构映射，不允许两套分离系统。

### 2.3 哲学层（长期演化）
- 风格不是装饰，而是信息结构的一部分。
- 设计系统的本质是“决策压缩器”：把重复决策前置成规则。
- 文档与代码同构更新，保证团队认知不漂移。

## 3. 风格方向

### 3.1 品牌气质关键词
- 梦幻
- 克制
- 神秘
- 温度

### 3.2 视觉语言
- 背景：深浅分层，不用纯黑纯白，保留呼吸感。
- 卡片：外扩填充 + 内边框 + 软阴影，形成轻悬浮秩序。
- 文本：标题偏叙事，正文偏信息，元信息偏弱提示。
- 强调：蓝青系为主，暖色用于情绪与提醒，不滥用高饱和红。

## 4. 主题模式规范（亮色 / 暗色）

### 4.1 模式策略
- 默认模式：暗色（匹配晨起低光环境）。
- 必备模式：亮色（满足白天长时阅读、截图传播、强光环境）。
- 实现原则：语义令牌映射双模式，不允许业务页面自己判断明暗后写特例色值。

### 4.2 双模式映射规则
- `backgroundPrimary / surface / card`: 控制层级深浅，不改变信息结构。
- `textPrimary / secondary / tertiary`: 保证在两模式下都满足阅读对比度。
- `accentInk / mood* / status*`: 保持语义一致，按模式调整亮度与饱和度。
- `stroke + shadow`: 亮色减阴影厚重感，暗色增加体积感。

### 4.3 可用性约束
- 两模式下核心文本均需可读，重点信息不可依赖单一颜色传达。
- 所有关键操作（按钮、状态、标签）在两模式下都要保持识别一致。

## 5. 令牌体系（Design Tokens）

### 5.1 色彩令牌
- `AppColor`: 背景、文本、强调、情绪、状态。
- 来源：`DS` 协议层统一输出，亮/暗动态映射。

### 5.2 排版令牌
- `AppTypography`: hero/screen/section/body/meta。
- `AppTextRole`: 文本语义角色，页面按角色取样式，不写裸字号。

### 5.3 布局令牌
- `AppSpacing`: xs/s/m/l/xl/xxl。
- `AppCornerRadius`: sm/md/lg/capsule。
- `AppCardStyle`: 外扩、内缩、边框透明度。
- `AppShadow`: softFloat/gentleLift 两层阴影系统。

## 6. 组件策略

### 6.1 组件分层
- 原子组件：标题、色块、标签、分隔线、统计块。
- 组合组件：梦境悬浮卡、操作按钮、底部导航预览。
- 页面层：按分区编排组件，不承载复杂业务状态。

### 6.2 组件约束
- 新页面优先用已有组件拼装，不新增同义组件。
- 组件输出语义明确，不允许“万能大组件”吞并多职责。
- 任何视觉调整先改令牌，再改组件，最后才改页面。

## 7. 动效与反馈

- 动效目标是“状态解释”，不是“装饰炫技”。
- 优先保留：页面进入、卡片浮层、关键状态切换。
- 减少连续大幅动效，避免晨起场景的感官负担。

## 8. 与各方对接协议

### 8.1 产品侧
- 需求文档标注组件语义名，不直接给十六进制颜色。
- 新页面先确认“复用哪些组件”和“新增哪些令牌”。

### 8.2 设计侧
- Figma 图层命名与令牌名保持同构。
- 交付同时包含亮暗两版关键态（默认、按压、禁用、空态）。

### 8.3 开发侧
- 开发按语义令牌实现，不直接硬编码关键视觉值。
- PR 必须说明是否涉及设计系统变更，涉及则同步文档。

### 8.4 运营与品牌侧
- 对外素材（截图、海报）优先沿用系统组件与文案层级。
- 视觉扩展不得破坏语义色约束。

## 9. 验收标准（MVP）

- 已具备亮色/暗色双模式可切换。
- 已具备可演示的设计系统页面（颜色、排版、材质、组件、导航）。
- 已形成一份可共享给多角色沟通的设计系统说明。
- 页面新增时不再重复定义基础视觉参数。

## 10. 迭代路线

### V1（当前）
- 双模式主题基础可用
- 令牌体系可复用
- 设计系统预览页可演示

### V2
- 加入动效令牌与状态机规范
- 加入图标、插画、空状态规范
- 增加可访问性（字号缩放、对比度）专项校验

### V3
- 连接业务真实页面（首页/统计/探索/我的）形成完整系统闭环
- 建立跨端映射策略（未来 Web/运营素材协同）
